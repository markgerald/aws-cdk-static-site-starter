# Repository Guidelines

## Project Structure & Module Organization

AWS CDK v2 TypeScript project for a static website on private S3, CloudFront OAC, ACM, and Cloudflare DNS.

- `bin/app.ts` is the CDK entry point; keep it limited to app composition and tagging.
- `lib/config/` resolves CDK context, `.env`, and environment defaults.
- `lib/stacks/`, `lib/constructs/`, and `lib/outputs/` contain stacks, reusable resource factories, and outputs.
- `lib/static-site-stack.ts` is a compatibility re-export for existing imports.
- `test/*.test.ts` contains Jest assertions for config, workflows, and CloudFormation templates.
- `website_src/` contains React/Vite source; `website_dist/` is generated by `npm run build:site`.
- `.github/workflows/ci.yml` runs only on pull requests touching `bin/`, `lib/`, `test/`, or `website_src/`, excluding Markdown-only changes. `deploy.yml` runs on pushes to `main`.

## Build, Test, and Development Commands

Use Node.js 22+.

- `npm ci` installs locked dependencies.
- `npm run build` compiles TypeScript with `tsc` into `dist/`.
- `npm run build:site` builds the React site into `website_dist/`.
- `npm run build:all` compiles CDK and builds the site.
- `npm test` runs the Jest CDK unit tests.
- `npm run test:coverage` runs Jest coverage.
- `npm run watch` starts TypeScript in watch mode.
- `npm run synth` runs `cdk synth` to generate CloudFormation templates.
- `npm run deploy` deploys all stacks; prefer `.env` copied from `.env.example`.

## Coding Style & Naming Conventions

TypeScript is strict (`strict`, `noImplicitAny`, `noImplicitReturns`, `alwaysStrict`). Use two-space indentation, single quotes, semicolons, and explicit interfaces for stack and factory props. Keep CDK construct IDs stable, such as `WebsiteBucket`, `Distribution`, and `OriginAccessControl`. Keep runtime configuration in `.env`, GitHub Actions variables, or `lib/config/`; do not hard-code account, region, or domain values.

## Testing Guidelines

Tests use Jest with `ts-jest` and match `**/test/**/*.test.ts`. Update assertions when resources, IAM policies, CloudFront behavior, website build behavior, or configuration defaults change. Before PRs, run `npm run build:all`, `npm test`, and `npm run synth`.

## Commit & Pull Request Guidelines

Use concise imperative commit subjects; conventional prefixes such as `fix(test): ...` are acceptable. PRs should describe infrastructure changes, list validation commands, link issues, and include screenshots or URLs when `website_src/` changes. Call out AWS, Cloudflare, ACM, or GitHub Actions variable changes.

## Security & Configuration Tips

Do not commit AWS credentials, Cloudflare tokens, generated `cdk.out/`, `website_dist/`, or local `.env` files. Keep ACM certificates for CloudFront in `us-east-1`. Start Cloudflare records as DNS only, and avoid adding Route 53 hosted-zone configuration unless the architecture changes intentionally.
